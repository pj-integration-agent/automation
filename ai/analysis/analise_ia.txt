| # | Nome do teste (log) | Motivo principal |
|---|---------------------|------------------|

---

**2. Motivos detalhados de cada falha**

### 2.1 Login inválido falhou
- **Expectativa:**  
  `locator('.error')` deveria conter a substring **“An internal error has occurred and has been logged.”**  
- **Resultado real:**  
  `<p class="error">The username and password could not be verified.</p>`
- **Análise:**  
  O teste foi escrito assumindo que o sistema exibe uma mensagem de *erro interno*. No entanto, na execução real, a aplicação retorna uma mensagem de *falha na autenticação* (“The username and password could not be verified.”).  
  As diferenças são:
  1. **Texto diferente** – não há nenhum “An internal error…” na página.  
  2. **Formato e pontuação** – a mensagem real tem ponto final e a palavra “verified” (não “verifi”).

### 2.2 Usuário deve atualizar telefone com campo vazio falhou
- **Expectativa:**  
  `locator('//*[@id="phone-error"]')` deveria conter a substring **“Phone is required.”**  
- **Resultado real:**  
  - Nenhum elemento com o XPath `//*[@id="phone-error"]` apareceu no DOM dentro dos 5 s de timeout.  
- **Análise:**  
  1. O XPath usado não corresponde a nenhum elemento existente (talvez o ID seja `phone-error` ou `phone-error` esteja em outro lugar).  
  2. Se o campo vazio deveria disparar um erro visível, o teste falhou antes mesmo de verificar o texto, porque a expectativa de localizador já não era satisfeita.  
  3. Pode também ser que o elemento exista, mas seja renderizado apenas após um evento (ex.: blur, submit), então o tempo de espera não foi suficiente.

---

**3. Sugestões de correção**

| # | Sugestões para o teste de login | Sugestões para o teste de telefone |
|---|--------------------------------|------------------------------------|
| **3.1** | - **Atualizar o texto esperado**: use a mensagem real retornada pela aplicação, por exemplo:<br>`await expect(page.locator('.error')).toContainText('The username and password could not be verified.')`<br>- Se quiser manter a mensagem “internal error”, verifique se o backend realmente dispara esse erro e, se não, retire a expectativa do teste.<br>- Para tornar o teste mais robusto, use <code>toHaveText</code> ou <code>toContainText</code> com regex que ignore variações de pontuação e maiúsculas, ex.:<br>`await expect(page.locator('.error')).toContainText(/username and password could not be verified/i)` | - **Usar um seletor CSS em vez de XPath** (mais rápido e mais legível):<br>`await expect(page.locator('#phone-error')).toContainText('Phone is required.')`<br>- **Adicionar um `waitForSelector`** antes de fazer a asserção, garantindo que o elemento exista e esteja visível:<br>`await page.waitForSelector('#phone-error', {state: 'visible', timeout: 8000})`<br>- Se o erro só aparece depois de submeter o formulário, assegure‑se de que a ação de submissão realmente foi executada (ex.: `await page.click('button[type="submit"]')`).<br>- Se o erro puder aparecer em outros formatos (ex.: `<span class="error">`), use <code>locator('text=Phone is required.')</code> que procura pelo texto independentemente do elemento. |
| **3.2** | - **Revisar a lógica de login**: Se o teste deve garantir que *erro interno* seja mostrado em algum cenário, crie um cenário de teste que force esse erro (ex.: mock de API retornando 500).<br>- **Adicionar logs detalhados** para entender o que a aplicação devolve quando há falha de login. | - **Verificar se o campo de telefone realmente dispara o erro**: após limpar o campo, certifique‑se de que a validação do lado do cliente/servidor está acionada.<br>- Se o erro aparece em um <code>div</code> com outra classe, atualize o seletor. Ex.:<br>`await expect(page.locator('.phone-error')).toContainText('Phone is required.')`<br>- **Usar `expect(page.locator('#phone-error')).toBeVisible()`** antes da verificação de texto para garantir que o elemento apareceu. |
